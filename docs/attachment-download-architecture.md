# Attachment Download Architecture

## Current Approach: Redirect via Storage Plugin URL

The attachment download endpoint (`GET /pages/{guid}/attachments/{filename}`) returns a JSON response containing a temporary download URL generated by the storage plugin:

```json
{ "url": "https://s3.../presigned-url", "contentType": "image/png", "filename": "photo.png" }
```

The frontend fetches this JSON, then uses the URL directly as the `<img>` src or download link. The browser loads the binary content straight from the storage backend.

### Why this approach

This avoids proxying binary data through API Gateway, which has multiple interacting issues (see below). The `getAttachmentUrl()` method on the storage plugin interface provides the abstraction — each plugin returns an appropriate URL for its backend (S3 presigned URL, GitHub raw URL, etc.).

## Why API Gateway Binary Proxying Is Problematic

We attempted three approaches to proxy binary data through API Gateway before settling on the URL redirect approach. Each hit a different limitation.

### Approach 1: Lambda `isBase64Encoded: true`

The Lambda base64-encodes the binary and returns it with `isBase64Encoded: true`. API Gateway is supposed to decode it back to raw bytes.

**Problem:** API Gateway only performs the decode when `BinaryMediaTypes` is configured on the REST API, AND the response `Content-Type` matches a configured type, AND the request `Accept` header also matches. Without all three conditions met, the client receives raw base64 text instead of binary.

### Approach 2: Configure `BinaryMediaTypes` on API Gateway

We added binary media types to the CDK REST API configuration.

- Setting `*/*` broke ALL endpoints — API Gateway treated every response (including JSON) as binary, returning 500 errors across the board.
- Setting specific types (`image/png`, `image/jpeg`, etc.) required the browser's `Accept` header to match. Axios sends `Accept: */*` by default, which doesn't match specific types. Even with a custom `Accept: image/*` header, content negotiation was fragile and difficult to debug.

### Approach 3: Base64 inside a JSON envelope

The Lambda returns the binary as a base64 string inside a normal JSON response:

```json
{ "data": "<base64-encoded-binary>", "contentType": "image/png" }
```

The frontend decodes the base64 and creates a Blob URL.

**Problem:** The Lambda used `fetch()` to download from the S3 presigned URL, then `response.arrayBuffer()` to get the bytes. The resulting Buffer had UTF-8 corruption — byte `0x89` (first byte of PNG header) was replaced with the UTF-8 replacement character `EF BF BD` (3 bytes). This corrupted the binary before base64 encoding even occurred.

**Potential fix (not implemented):** Use the S3 SDK `GetObjectCommand` directly instead of `fetch()` with a presigned URL. The SDK returns a proper binary stream that wouldn't suffer from UTF-8 text interpretation. This would make the JSON envelope approach viable if we need to revisit it.

### Summary of limitations

| Approach | Failure mode |
|---|---|
| `isBase64Encoded: true` | Requires `BinaryMediaTypes` + matching `Accept` header |
| `BinaryMediaTypes: */*` | Breaks all JSON endpoints |
| `BinaryMediaTypes: specific types` | Fragile content negotiation with axios |
| Base64 in JSON envelope | `fetch()` in Lambda corrupts binary via UTF-8 interpretation |
| **URL redirect (current)** | Works reliably, storage-agnostic via plugin interface |

## Future Considerations

- If a storage plugin cannot generate download URLs (e.g., a database-backed plugin), the JSON envelope approach can be revisited by using the storage SDK directly to read binary (bypassing `fetch()`).
- API Gateway HTTP APIs (v2) handle binary responses more naturally than REST APIs (v1). Migrating would simplify binary proxying if needed.
- Lambda Function URLs also support binary responses without the `BinaryMediaTypes` configuration dance.
